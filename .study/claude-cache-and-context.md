# Claude API - 캐시 토큰과 컨텍스트 윈도우

## 1. 컨텍스트 윈도우란?

Claude가 한 번에 읽을 수 있는 **대화 내용의 최대 크기**.

- Claude Opus 4.6: **1M (100만 토큰)** ≈ 약 75만 단어
- Claude Sonnet 4.6: **200K (20만 토큰)**
- Claude Haiku 4.5: **200K (20만 토큰)**

### 매 턴마다 보내는 전체 내용

```
┌─────────────────────────────────────────┐
│ 시스템 프롬프트 (지시사항, 규칙)            │  ← 매번 포함
│ 도구 정의 (Bash, Read, Write, Agent 등)   │  ← 매번 포함
│ 대화 1턴 (사용자 질문 + AI 응답)           │  ← 매번 포함
│ 대화 2턴                                 │  ← 매번 포함
│ ...                                      │
│ 대화 N턴 (현재 사용자 질문)                │  ← 새 메시지
└─────────────────────────────────────────┘
          ↑ 이 전체가 컨텍스트 윈도우 안에 들어가야 함
```

**중요**: Claude는 이전 대화를 "기억"하는 게 아니라, **매번 전체 대화를 처음부터 다시 읽는다.**

## 2. 캐시 토큰이란?

매번 전체 대화를 다시 처리하면 느리고 비싸기 때문에, **이미 처리한 부분을 캐시에 저장**해두는 최적화.

### 토큰 종류 4가지

| 토큰 종류 | 의미 | 과금 |
|-----------|------|------|
| `input_tokens` | 캐시에 없어서 새로 처리한 토큰 | 100% |
| `output_tokens` | Claude가 생성한 응답 토큰 | 100% |
| `cache_creation_input_tokens` | 최초 캐시에 저장할 때 | **125%** (약간 비쌈) |
| `cache_read_input_tokens` | 캐시에서 재사용한 토큰 | **10%** (90% 할인) |

### 작동 흐름 예시

```
=== 턴 1 ===
시스템 프롬프트 (5만 토큰) → 처음이라 캐시 생성 → cache_creation: 50,000
사용자 질문 (100 토큰)     → 새 입력            → input: 100
AI 응답 (500 토큰)         → 생성               → output: 500

=== 턴 2 ===
시스템 프롬프트 (5만 토큰) → 캐시에 있음!       → cache_read: 50,000 ★
턴1 대화 (600 토큰)        → 캐시에 추가 저장    → cache_creation: 600
사용자 새 질문 (150 토큰)   → 새 입력            → input: 150
AI 응답 (800 토큰)         → 생성               → output: 800

=== 턴 3 ===
시스템 + 턴1 + 턴2         → 캐시에 있음!       → cache_read: 51,400 ★
사용자 새 질문 (200 토큰)   → 새 입력            → input: 200
AI 응답 (600 토큰)         → 생성               → output: 600
```

### 캐시 만료

- **5분 캐시**: 일반적인 대화 턴 간 캐시 (ephemeral_5m)
- **1시간 캐시**: 시스템 프롬프트 등 고정 내용 (ephemeral_1h)
- 만료되면 다시 `cache_creation` 발생 → 약간의 비용 추가

## 3. 컨텍스트 윈도우가 꽉 차면?

### 3-1. 압축 (Compression)이 발생

대화가 길어져서 컨텍스트 윈도우 한계에 가까워지면, **오래된 대화를 요약/압축**한다.

```
=== 정상 상태 (60만 토큰) ===
시스템 프롬프트          10만
도구 정의               5만
대화 1~30              30만  ← 원본 유지
대화 31~40             15만  ← 원본 유지

=== 한계 도달 (100만 토큰 초과) ===
시스템 프롬프트          10만  ← 유지
도구 정의               5만   ← 유지
[대화 1~30 요약]        3만   ← ★ 30만 → 3만으로 압축됨
대화 31~50             25만  ← 원본 유지
새 메시지               5만   ← 원본 유지
```

### 3-2. 압축으로 인해 손실되는 정보

| 원본 대화 | 압축 후 |
|-----------|---------|
| "src/parsers/jsonl-parser.ts의 37번째 줄에서 `getProjectsDir` 함수의 인자를 `claudeDir: string`으로 변경" | "jsonl-parser.ts의 함수를 수정" |
| "Docker 빌드 시 `npm install --omit=dev`로 프로덕션 의존성만 설치" | "Docker 설정 변경" |
| 코드 블록 전체 내용 | 삭제되거나 간략 설명으로 대체 |

### 3-3. 이로 인한 문제들

#### 환각 (Hallucination)
```
사용자: "아까 만든 함수 이름이 뭐였지?"
Claude: "getSessionData 함수를 만들었습니다"  ← 실제로는 getSessionMessages였음
```
압축된 요약에 정확한 함수명이 없어서 그럴듯한 이름을 만들어냄.

#### 맥락 단절
```
사용자: "아까 말한 그 버그 수정해줘"
Claude: "어떤 버그를 말씀하시는지 구체적으로 알려주세요"
```
초반 대화가 압축되어 버그의 구체적 내용을 모름.

#### 중복 작업
```
이미 초반에 만든 유틸 함수를 잊고 같은 기능의 함수를 다시 만듦
```

## 4. 캐시 ≠ 컨텍스트 윈도우

이 둘은 완전히 별개의 개념이다.

```
┌──────────────────────────────────────────────────┐
│                컨텍스트 윈도우 (1M)                 │
│  "Claude가 읽을 수 있는 대화의 최대 크기"            │
│                                                    │
│  ┌────────────────┐  ┌──────────────────────┐     │
│  │ 캐시된 부분      │  │ 새로 처리하는 부분     │     │
│  │ (빠르고 저렴)    │  │ (느리고 비쌈)         │     │
│  └────────────────┘  └──────────────────────┘     │
│                                                    │
│  캐시는 이 안에서의 "처리 속도 최적화"일 뿐          │
│  윈도우 크기를 늘려주지 않는다                       │
└──────────────────────────────────────────────────┘

윈도우 밖 (잘려나간 대화) → 캐시에 있어도 보내지 않음
```

| | 캐시 | 컨텍스트 윈도우 |
|---|---|---|
| 역할 | 처리 속도/비용 최적화 | 대화 내용의 기억 용량 |
| 한계 | 시간 기반 만료 (5분~1시간) | 토큰 수 제한 (1M) |
| 꽉 차면 | 캐시 미스 → 다시 계산 (느려질 뿐) | 압축 발생 → 정보 손실 |
| 환각 유발 | X | O |

## 5. 실제 데이터로 보는 예시

이 세션(ai-usage-visualization 프로젝트)의 실제 토큰 사용량:

```
input_tokens:              9,183      ← 캐시에 없는 새 토큰
output_tokens:            45,791      ← Claude가 생성한 토큰
cache_creation:          503,834      ← 캐시 최초 저장
cache_read:           10,153,307      ← 캐시에서 재사용

실제 API 호출 토큰:       54,974  (input + output)
캐시 포함 전체:       10,712,115  (전부 합산)
```

cache_read가 1000만이 넘는 이유: 350+개 메시지 동안 매 턴마다 시스템 프롬프트 + 전체 대화를 다시 보내지만, 캐시 덕분에 10% 비용으로 처리됨.

## 6. 대화가 길어질 때 대응 방법

1. **새 세션 시작**: 컨텍스트가 60~70% 차면 새 대화를 시작
2. **메모리 시스템 활용**: CLAUDE.md나 memory에 중요 정보를 저장 → 새 세션에서도 참조 가능
3. **구체적으로 참조**: "아까 그거" 대신 파일명, 함수명을 직접 언급
4. **작업 단위 분리**: 하나의 큰 작업을 여러 세션으로 나누기
